GetSellerList

Use this call to get a list of the items posted by the authenticated user, along with the related item data.

GetSellerList returns an array of zero, one, or multiple items, depending on the user's activity and the inputs used in the call. You can use various input values in your GetSellerList request to either increase or filter the data returned. For example, you can include a date range to restrict the items returned, or you can specify different DetailLevel values to control the fields contained in the response.

See Browsing a Seller's Items for a detailed description of retrieving items using GetSellerList.

Request Details

GetSellerList returns a maximum of 5,000 items per call (this is possible if you do not specify a Detail Level, and thus return the briefest set of data for each item). However, if you specify any DetailLevel value (and thus increase the fields returned for each item), you might need to use Pagination values to obtain the complete result set. If you use DetailLevel, the response can contain no more than 200 items per call.

Many item details (such as item descriptions and listing enhancements) don't change frequently, and retrieving these values adds to the overhead of your calls. After you initially download the details for a set of items, subsequent requests for those items should retrieve only the item information that is likely to change. Because of this, it is best to avoid using a DetailLevel of ReturnAll or a GranularityLevel of Coarse with GetSellerList. Alternatively, consider using GetSellerEvents instead of GetSellerList.

If you need to use ReturnAll to retrieve all details for a set of items, use a smaller EntriesPerPage value to limit the number of items returned from the call. In addition, specify shorter time ranges with EndTimeFrom and EndTimeTo to increase performance.

You can specify either a DetailLevel or a GranularityLevel in your request, but not both. See Specifying Detail Levels and Granularity Levels for more information on setting the detail level for an API call.

Use the Pagination input property to control the pages of data returned from your call. Pagination.EntriesPerPage controls how many items are returned per call and Pagination.PageNumber specifies which page of data to retrieve in the current call. The value specified for EntriesPerPage must be between 1 and 200.

Note that if you set DetailLevel to ItemReturnDescription or ReturnAll, then you must also set pagination values in your request.

You can use multiple date-range filters in the same call. For example, use the StartTimeFrom and StartTimeTo input properties to return items where the listing started between the two specified dates. Use EndTimeFrom and EndTimeTo to return items where the listing ends between the two specified dates.

Date-range filters are individually optional, but at least one date-range filter must be specified if pagination is used (applicable to all detail levels). The reverse is not true, a call can use a date-range filter without also specifying pagination values.

Working with the Response

GetSellerList returns a list of the sellers items in an ItemArray container.

  1. Traverse the response and visit each ItemType object contained in the ItemArray container.
  2. Inspect the properties of each ItemType object to get the values for the associated listing.
  3. Store the pertinent item data, as needed by the application.

In the output, the ReturnedItemCountActual property indicates how many items were returned by the call; Count indicates how many items could be returned. HasMoreItems, if true, indicates that there are more items to be returned, requiring one or more additional calls. PageNumber indicates the page of data returned and ItemsPerPage the number of items returned per call.

Testing GetSellerList

You can test this call in the Sandbox environment. To test the call, add at least two or three new items with a test user using AddItem.

Some data values are only returned if the call requestor is the item seller. To test the call from the perspective of another user, execute GetSellerList using a user other than the one who listed the item.

For applications that list different types of items (such as Chinese auctions and fixed-price items), create all possible types of listings using AddItem, then attempt to retrieve the data with GetSellerList.

If applicable to the application's run-time use cases, list test items with flat and calculated shipping rates. Then retrieve these with GetSellerList.

For applications that create competitive-bid item listings (Chinese auctions), bid on the items using the Sandbox user interface before retrieving the item using GetSellerList. This will populate the high bidder data for items in the result set.

Test calls to GetSellerList with different combinations of detail level, pagination, and date-based filtering to cover all of the possible scenarios the application might have to accommodate under real world usage conditions.

  • GetBidderList - Retrieves all items the user is currently bidding on, and the ones they have won or purchased.

  • ReviseItem - Enables a seller to change the properties of a currently active listing.
